### BUG-M-001 数据库连接缺少 charset 配置导致中文乱码

| 项目 | 内容 |
|------|------|
| **严重程度** | 🔴 严重 (Major) |
| **模块** | 后端 - db.js |
| **状态** | ✅ 已修复 |
| **修复版本** | V2.3.1 |

**问题描述**：
`server/src/db.js` 中 mysql2 连接池未设置 `charset: 'utf8mb4'`。MySQL 的 utf8 字符集仅支持 3 字节 UTF-8 编码，而中文字符中的许多扩展字符需要 4 字节编码（utf8mb4）。所有通过 API 写入的含中文数据均被存储为乱码。

**复现步骤**：
1. 启动服务器
2. 通过 API 注册用户：`POST /api/auth/register` body 含中文 nickname/bio/goals/skills/interests
3. 通过 API 创建中文标题的日程
4. 查看数据库：所有中文字段显示为 `�����` 乱码

**实际结果**：
```json
// API 返回的数据
{"nickname":"�����ǳ�","bio":"�Ȱ���̵Ĵ�ѧ��","goals":"ͨ����ĩ����"}
```
数据库 hex 值为非标准 UTF-8 编码（Latin-1 错误解释的 UTF-8 字节）。

**根因分析**：
`server/src/db.js` 第 3-12 行，连接池配置缺少 `charset` 参数：
```javascript
const pool = mysql.createPool({
  // ... 其他配置
  timezone: 'Z'
  // 缺少: charset: 'utf8mb4'
})
```

mysql2 的默认 charset 为 `UTF8_GENERAL_CI`，在 MySQL 中对应 3 字节 UTF-8（别名 `utf8`），不支持中文 CJK 扩展区字符。

**修复措施**：
在连接池配置中添加 `charset: 'utf8mb4'`：
```javascript
const pool = mysql.createPool({
  // ...
  charset: 'utf8mb4',  // 新增
  timezone: 'Z'
})
```

**修复验证**：
通过 Node.js 原生 http 模块发送 UTF-8 编码请求，中文数据正确存储和读取。
- 修复前：nickname 显示 `�����ǳ�`
- 修复后：nickname 显示 `中文昵称`

**影响范围**：
- 所有含中文的用户数据：昵称、个人简介、近期目标、技能标签、兴趣爱好
- 所有含中文的日程标题、备注、地点
- 所有含中文的动态内容
- AI 简历生成的个人概述和成长洞察（读取乱码数据后输出质量下降）

**已存在乱码数据处理**：
修复前已存储的乱码数据无法自动恢复，需手动清理或重新录入。

### BUG-M-003 Express 未配置信任代理导致速率限制无法识别客户端 IP

| 项目 | 内容 |
|------|------|
| **严重程度** | 🔴 严重 (Major) |
| **模块** | 后端 - index.js |
| **状态** | 🔧 待修复 |

**问题描述**：
`server/src/index.js` 中 Express 应用未启用 `app.set('trust proxy', 1)`。在生产环境中，Nginx 反向代理转发请求时，Express 看到的 `req.ip` 始终是 `127.0.0.1` 而非真实客户端 IP。这会导致：
1. 安全日志无法记录真实 IP
2. 速率限制（如添加）无法区分不同用户
3. IP 白名单/黑名单功能失效

**代码位置**：`server/src/index.js` 第 27 行（`const app = express()` 之后缺少配置）

**修复建议**：
```javascript
const app = express()
app.set('trust proxy', 1)  // 信任第一层代理
```

**影响范围**：生产环境（Nginx 反向代理后），开发环境无影响。

### BUG-N-001 服务器启动时数据库连接未预验证

| 项目 | 内容 |
|------|------|
| **严重程度** | 🟡 一般 (Medium) |
| **模块** | 后端 - db.js / index.js |
| **状态** | 🔧 待修复 |

**问题描述**：
mysql2 的 `createPool()` 采用惰性连接模式——连接池创建时不会立即验证数据库连接。如果数据库凭据错误或 MySQL 未启动，服务器仍然"成功启动"（HTTP 端口正常监听），但首次数据库查询会失败并抛出异常。

虽然有全局错误处理中间件防止进程崩溃，但用户在首次请求时才收到 500 错误，体验不佳。

**复现步骤**：
1. 在 `.env` 中配置错误的 DB_PASSWORD
2. 启动服务器 → 成功启动（仅 DB Config 日志正常）
3. 请求 `GET /api/schedules` → 返回 500

**修复建议**：
在 `index.js` 启动监听之前添加健康检查查询：
```javascript
// 在 app.listen 之前
db.query('SELECT 1').then(() => {
  console.log('[DB] Connection verified')
}).catch(err => {
  console.error('[FATAL] Database connection failed:', err.message)
  process.exit(1)
})
```

**影响**：首次请求失败时用户困惑，调试时无法快速定位数据库连接问题。

### BUG-L-001 JWT_SECRET 默认值缺少强制覆盖机制

| 项目 | 内容 |
|------|------|
| **严重程度** | 🟢 轻微 (Minor) |
| **模块** | 后端 - index.js / auth.js / middleware/auth.js |
| **状态** | 🔧 待修复 |

**问题描述**：
当 `JWT_SECRET` 未配置或使用默认值 `replace-with-strong-secret` 时，系统仅输出一行 console.warn 警告，然后静默回退到 `'dev-secret'`。任何人都能使用 `'dev-secret'` 伪造有效的 JWT Token。

**代码位置**：
- `server/src/index.js:16-19`
- `server/src/routes/auth.js:11`
- `server/src/middleware/auth.js:11`

**修复建议**：
生产环境检测到默认 secret 时应拒绝启动：
```javascript
if (!process.env.JWT_SECRET || process.env.JWT_SECRET === 'replace-with-strong-secret') {
  if (process.env.NODE_ENV === 'production') {
    console.error('[FATAL] JWT_SECRET must be configured for production.')
    process.exit(1)
  }
  console.warn('[WARN] Using insecure default JWT_SECRET. Set a strong secret in .env for production.')
  process.env.JWT_SECRET = 'dev-secret'
}
```

**影响**：任何知道默认 secret 的人都可以伪造 JWT Token 访问任意用户的数据。

### BUG-L-002 微信登录失败时暴露第三方 API 错误详情

| 项目 | 内容 |
|------|------|
| **严重程度** | 🟢 轻微 (Minor) |
| **模块** | 后端 - routes/auth.js |
| **状态** | 📋 计划中 |

**问题描述**：
`POST /api/auth/wechat` 中，当微信 API 返回错误时，直接将 `data.errmsg` 透传给客户端：
```javascript
if (data.errcode) return res.status(502).json({ message: data.errmsg })
```

这会将微信平台的错误信息（可能包含内部细节）暴露给前端用户。

**修复建议**：
返回通用错误消息，将详细信息记录到服务器日志：
```javascript
if (data.errcode) {
  console.error('[WeChat API Error]', data.errcode, data.errmsg)
  return res.status(502).json({ message: 'WeChat login failed. Please try again.' })
}
```

**影响**：可能暴露第三方 API 的内部错误信息和错误码。
